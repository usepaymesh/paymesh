## `createClient`

Every Paymesh installation starts from `createClient`. This is the composition point where you decide:

- which provider powers the runtime
- whether data is persisted locally
- how built-in tables are named
- whether plugins extend routes, hooks, and schema
- whether normalized objects should carry raw provider payloads
- which origins are allowed for checkout redirect URLs

```ts title="src/lib/paymesh.ts"
import { createClient } from "paymesh";
import { postgres } from "@paymesh/postgres";
import { stripe } from "@paymesh/stripe";
import { auditLog } from "@paymesh/audit-logs";
import { dash } from "@paymesh/dash";

export const paymesh = createClient({
  provider: stripe({
    secret: "sk_test_123",
    webhookSecret: "whsec_123",
  }),
  database: postgres("postgres://postgres:postgres@localhost:5432/paymesh", {
    persistRaw: true,
  }),
  includeRaw: false,
  schema: {
    prefix: "paymesh",
  },
  plugins: [
    auditLog({
      events: ["payment.*", "customer.*"],
    }),
    dash({
      path: "/internal/paymesh",
      auth: async ({ request }) => {
        const session = await auth({ request });
        return {
          allowed: session.user.role === "admin",
          actor: {
            id: session.user.id,
            email: session.user.email,
            name: session.user.name,
          },
        };
      },
    }),
  ],
});
```

## Option reference

| Option | Required | Default | What it controls |
| --- | --- | --- | --- |
| `provider` | yes | none | The provider implementation created by packages such as `@paymesh/stripe` or `@paymesh/polar`. |
| `database` | no | `undefined` | Persists normalized data, enables migrations, webhook deduplication, catalog sync, plugins that require storage, and operational tooling. |
| `includeRaw` | no | `false` | Propagates raw provider payloads through client reads and webhook events when enabled. |
| `trustedOrigins` | no | `undefined` | Allowlist of origins accepted for checkout redirect URLs. |
| `sandbox` | no | `undefined` | Asserts sandbox mode matches the provider. Throws if the provider reports a different mode. |
| `schema` | no | generated defaults | Renames built-in tables, changes the prefix, and adds extra columns or custom tables. |
| `plugins` | no | `[]` | Registers plugin routes, hooks, schema extensions, setup logic, and runtime client extensions. |

## `provider`

The provider is the only required option. It supplies:

- capabilities such as `pix`, `customers`, `webhooks`, `subscriptions`, and `customerPortal`
- normalized payment and customer methods
- webhook verification and event mapping
- optional dashboard helpers and catalog sync

Pick a single provider per client instance. If you need multiple providers, create multiple clients and compose them one level above Paymesh.

```ts title="src/lib/paymesh.ts"
import { createClient } from "paymesh";
import { polar } from "@paymesh/polar";

export const paymesh = createClient({
  provider: polar({
    accessToken: "polar_access_123",
    webhookSecret: "whsec_123",
  }),
});
```

### Provider capability checks

Paymesh exposes provider capabilities on the runtime so your product code can branch only when necessary.

```ts title="src/server/capabilities.ts"
if (paymesh.capabilities.pix) {
  const pix = await paymesh.pix.create({
    amount: 5900,
    currency: "BRL",
    description: "Invoice #42",
  });

  return pix.copyPasteCode;
}
```

## `database`

Without a database, Paymesh still works for direct payment creation and webhook normalization. With a database, it becomes operationally useful.

Adding `database` enables:

- normalized persistence for `customers`, `checkouts`, `pix`, `subscriptions`, `products`, `prices`, and `webhookEvents`
- webhook delivery acquisition and deduplication
- catalog push and migration workflows from `@paymesh/cli`
- plugins that declare `config.database: true`
- dashboard views powered by stored data instead of live provider fetches

```ts title="src/lib/paymesh.ts"
import { createClient } from "paymesh";
import { postgres } from "@paymesh/postgres";
import { stripe } from "@paymesh/stripe";

export const paymesh = createClient({
  provider: stripe({
    secret: "sk_test_123",
    webhookSecret: "whsec_123",
  }),
  database: postgres("postgres://postgres:postgres@localhost:5432/paymesh", {
    persistRaw: true,
  }),
});
```

<Callout type="info">
  `persistRaw` is configured on the database adapter, not on `createClient`. When it is enabled, stored rows retain provider payloads for reconciliation and debugging.
</Callout>

## `sandbox`

The `sandbox` option in `createClient` is an **assertion**. It does not change the provider's sandbox mode â€” it validates that the provider matches your expectation.

```ts title="src/lib/paymesh.ts"
createClient({
  provider: stripe({
    secret: "sk_test_123",
  }),
  sandbox: true, // passes â€” provider auto-detected sandbox
});
```

If the assertion fails, the client throws:

```ts
createClient({
  provider: stripe({
    secret: "sk_live_123",
  }),
  sandbox: true, // PaymeshError: client sandbox does not match provider
});
```

This catches accidental production usage in development environments.

### Per-call sandbox override

When a database is configured, read methods also accept a per-call `sandbox` override:

```ts title="src/server/admin.ts"
// Override sandbox for a single read
const customer = await paymesh.customers.get("cus_123", {
  sandbox: false,
});

const page = await paymesh.customers.list({
  sandbox: true,
});
```

The per-call option is available on `customers.get`, `customers.list`, and `pix.get`.

## `includeRaw`

`includeRaw` controls runtime objects, not storage. When enabled, return values and webhook events include the original provider payload in `raw`.

```ts title="src/server/raw.ts"
const client = createClient({
  provider: stripe({
    secret: "sk_test_123",
    webhookSecret: "whsec_123",
  }),
  includeRaw: true,
});

const pix = await client.pix.create({
  amount: 9900,
  currency: "BRL",
  description: "Invoice #99",
});

console.log(pix.raw);
```

Use `includeRaw` when:

- you are incrementally migrating from a provider SDK and still need provider-specific fields
- support tooling needs the upstream payload at request time
- you are debugging webhook mappings

Keep it off when you want application code to stay strict about the normalized contract.

## `trustedOrigins`

`trustedOrigins` lets you restrict which origins Paymesh will accept in checkout redirect fields such as `successUrl`, `cancelUrl`, and `returnUrl`.

```ts title="src/lib/paymesh.ts"
createClient({
  provider: stripe({
    secret: "sk_test_123",
  }),
  trustedOrigins: [
    "https://app.example.com",
    "http://localhost:3000",
  ],
});
```

Rules:

- entries must be absolute origins only
- direct `payments.create()` calls must still pass absolute redirect URLs
- when `trustedOrigins` is configured, direct checkout calls are rejected if a redirect URL points at an untrusted origin
- relative redirect URLs are only supported inside request-backed plugin routes that explicitly resolve them against the current request

### What gets checked

Paymesh validates these fields when present:

- `successUrl`
- `cancelUrl`
- `returnUrl`

If any of them is absolute, its origin must be in `trustedOrigins`.
If any of them is relative, Paymesh rejects it in direct client calls and only allows it when a request-backed route resolves it first.

### Practical examples

```ts title="src/server/checkout.ts"
// Allowed: absolute URL on a trusted origin.
await paymesh.payments.create({
  amount: 5900,
  currency: "USD",
  successUrl: "https://app.example.com/success",
});

// Rejected: relative URL from direct app code.
await paymesh.payments.create({
  amount: 5900,
  currency: "USD",
  successUrl: "/success",
});
```

```ts title="src/server/dash.ts"
// Allowed inside a request-backed route when Dash resolves the URL.
const successUrl = context.resolveTrustedUrl("/success");
```

### Failure modes

Paymesh throws `invalid_request` when:

- a `trustedOrigins` entry is not an origin-only URL
- a redirect URL points at an origin that is not trusted
- a relative redirect URL is used without a request-backed route to resolve it

## `schema`

`schema` lets you control physical persistence details without rewriting the client contract.

```ts title="src/lib/paymesh.ts"
export const paymesh = createClient({
  provider: stripe({
    secret: "sk_test_123",
    webhookSecret: "whsec_123",
  }),
  schema: {
    prefix: "billing",
    tables: {
      customers: {
        name: "billing_customers",
        fields: {
          workspaceId: {
            type: "string",
            required: true,
            index: true,
          },
        },
      },
    },
  },
});
```

### Schema responsibilities

| Schema area | What it affects |
| --- | --- |
| `prefix` | Default prefix for all built-in tables. |
| `tables` | Built-in table name overrides plus extra fields for those tables. |
| `customTables` | Additional custom tables, commonly contributed by plugins. |

### What schema does not do

- it does not change provider API behavior
- it does not alter webhook mapping semantics
- it does not change the normalized Paymesh runtime method signatures

## `plugins`

Plugins are registered in an ordered array. During client creation they can:

- validate runtime requirements
- extend the resolved schema
- add HTTP routes
- attach middleware
- react to built-in webhook events
- define custom plugin events
- expose extra runtime APIs through `extends`

```ts title="src/lib/paymesh.ts"
import { createClient } from "paymesh";
import { auditLog } from "@paymesh/audit-logs";
import { dash } from "@paymesh/dash";
import { postgres } from "@paymesh/postgres";
import { stripe } from "@paymesh/stripe";

export const paymesh = createClient({
  provider: stripe({
    secret: "sk_test_123",
    webhookSecret: "whsec_123",
  }),
  database: postgres("postgres://postgres:postgres@localhost:5432/paymesh"),
  plugins: [
    auditLog(),
    dash({
      path: "/internal/paymesh",
      auth: async ({ request }) => {
        const session = await auth({ request });
        return {
          allowed: session.user.role === "admin",
        };
      },
    }),
  ],
});
```

### Plugin ordering

Plugin registration order matters when:

- multiple plugins observe the same hook
- routes overlap
- setup logic expects another plugin to have extended the schema first

Keep official operational plugins near the end of the array unless you have a strong reason not to.

## Runtime shape after composition

After `createClient`, the runtime typically exposes:

- `paymesh.provider`
- `paymesh.capabilities`
- `paymesh.payments`
- `paymesh.pix`
- `paymesh.customers`
- `paymesh.webhooks`
- `paymesh.routes`
- `paymesh.plugins`
- `paymesh.database`
- `paymesh.schema`
- plugin-contributed extensions

That means the client becomes the single integration boundary for the rest of your application.

## Recommended composition pattern

Create one server-only module that exports the configured client, and import that module everywhere else.

Do not:

- instantiate providers inside route handlers
- scatter schema options across unrelated files
- assemble plugin lists conditionally in multiple places

Do:

- keep one authoritative `paymesh.ts`
- colocate migration and webhook examples with that file
- treat `createClient` as application infrastructure, not request-time logic
