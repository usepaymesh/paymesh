## Providers are implementations, not the product API

Provider packages implement the core contracts exported by `paymesh`.

That means a provider is responsible for:

- payment creation
- PIX support when available
- customer operations
- catalog sync
- dashboard affordances
- webhook verification
- raw event mapping into normalized events

## Current first-party providers

### Stripe

Stripe currently implements:

- payments
- PIX
- customers
- catalog
- dashboard helpers
- webhook verification and event mapping

### Polar

Polar currently implements:

- payments
- customers
- catalog
- dashboard helpers
- webhook verification and event mapping

Polar does **not** expose PIX in the current repository.

### AbacatePay

AbacatePay currently implements:

- checkout payments (catalog-driven with product IDs)
- PIX transparent charges
- customers
- catalog reads
- webhook verification and event mapping

AbacatePay does **not** expose subscriptions, refunds, or customer portal in the current repository.

### Dodo

Dodo currently implements:

- hosted checkout payments through Dodo product carts
- customers
- catalog reads
- subscription-aware dashboard sync
- webhook verification and event mapping

Dodo does **not** expose `paymesh.pix` in the current repository. BRL hosted checkouts can still present Pix inside the Dodo checkout page, but the provider does not currently return a native QR-code-first PIX object.

## Capability flags

Provider differences are carried through capability flags rather than hidden.

This is important for product code because it lets you ask:

- does this provider support PIX?
- does this provider support customers?
- does this provider support subscriptions?

without guessing from package name or trial-and-error.

## Provider-specific inputs that remain explicit

Some provider-specific differences still matter. For example:

- Polar requires `productIds` for checkout creation
- AbacatePay requires `productIds` for checkout creation and uses catalog-driven checkout
- Dodo requires `productIds` for payment creation and can enable hosted Pix inside BRL checkouts
- Stripe PIX supports `amountIncludesIof` and expiration settings
- dashboard integrations vary by provider

Paymesh keeps those differences at the edges instead of forcing one fake "universal" shape where it would be misleading.
